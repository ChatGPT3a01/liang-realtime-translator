先看全貌：拍一張照片，中間發生什麼事

看懂這條路，後面每一步就知道自己在做哪一段
  1. 使用者按「📷 拍照翻譯」→ 手機打開後鏡頭拍照
  2. 前端先把照片縮小（省流量、變快）
  3. 前端把縮小的照片送到後端的 /api/vision
  4. 後端呼叫 providers.analyze()，把圖丟給 Gemini 讀
  5. Gemini 回傳 {summary, translation}（摘要＋翻譯）
  6. 前端把摘要與翻譯顯示在畫面上
👉 這一章就是照著這 6 步，一段一段做出來。

白話觀念：什麼是「多模態」

AI 同時看圖 + 讀字

一般翻譯只吃「文字」。多模態（multimodal）的 AI 能同時理解「圖片」和「文字」——所以它能看懂照片裡的菜單、招牌。

你可以把它想成：以前的 AI 只會「聽你講」，多模態的 AI 還會「用眼睛看」。

🔎 我們用的 gemini-3.5-flash 就會看圖與 PDF，所以直接把照片丟給它就行，不必自己做文字辨識（OCR）。

步驟一：教後端「讀圖」

在 providers.py 新增 analyze()
步驟 1／4
🎯 這一步要做什麼：寫一個函式 analyze()，收到一張圖，交給 Gemini 看，請它回「摘要＋翻譯」。

📂 打開這個檔案：providers.py，把 AI 給你的函式貼在檔案最下面（其他函式的後面）。

💬 對你的 AI 助手這樣說（複製整段貼過去）：

請在 providers.py 新增一個 analyze() 函式：
參數有 api_key、target_name、text、file_bytes、mime_type。
用 Gemini 的 gemini-3.5-flash 讀圖（file_bytes）或讀文字（text），
要求它只回 JSON：{"summary": 用 target_name 語言摘要, "translation": 完整翻譯}。
記得把 thinking 關掉（thinking_budget=0），避免 JSON 被截斷，最後做容錯解析。

步驟一：AI 會給你這段（貼上去）

貼進 providers.py
# providers.py — 貼在檔案最下面
# file_bytes（圖片/PDF）或 text（純文字）擇一給
def analyze(api_key, target_name, text=None, file_bytes=None, mime_type=None):
    from google import genai
    from google.genai import types
    client = genai.Client(api_key=api_key)
    system = (f"Read the input. Write a concise summary in {target_name}, "
              f"and a faithful full translation into {target_name}. "
              'Respond ONLY as JSON: {"summary":"...","translation":"..."}')
    parts = []
    if file_bytes is not None:
        parts.append(types.Part.from_bytes(data=file_bytes, mime_type=mime_type))
    if text:
        parts.append(text)
    resp = client.models.generate_content(
        model='gemini-3.5-flash',
        contents=parts,
        config=types.GenerateContentConfig(system_instruction=system,
            response_mime_type='application/json', max_output_tokens=8192,
            thinking_config=types.ThinkingConfig(thinking_budget=0)))   # 關閉思考
    return parse_json(resp.text)   # 容錯解析 summary/translation

步驟一：逐段看懂

每一段在做什麼，白話講一遍
這幾行在做什麼
client = genai.Client(...)拿你的金鑰，建立跟 Gemini 講話的連線
system = (...)給 AI 的指令：讀輸入，用你的語言寫摘要＋翻譯，只准回 JSON
if file_bytes ...：Part.from_bytes把「圖片」包成 Gemini 看得懂的格式
if text：parts.append(text)如果是純文字（CH12 會用到），就直接放文字
thinking_budget=0叫 AI「別想太多、直接答」——這行很關鍵，下一張說明
parse_json(resp.text)把 AI 回的文字解析成 {summary, translation}

步驟一 · 踩過的雷：JSON 被截斷

為什麼一定要關 thinking

Gemini 3.x flash 預設會先「思考」，思考會吃掉輸出額度。內容一長，答案的 JSON 就被切一半（少了結尾的 }），程式一解析就壞。

✅ 解法：thinking_budget=0 關掉思考（讀圖翻譯本來就不用推理），輸出更穩也更快；再加「容錯解析」把被切斷的內容救回來。
🔧 卡住了？　若結果常常是「（無摘要）」或報 JSON 錯誤 → 十之八九就是沒關 thinking，或 max_output_tokens 太小。

步驟二：開一個「收照片」的門

在 app.py 新增 /api/vision
步驟 2／4
🎯 這一步要做什麼：做一個網址 /api/vision，讓前端能把照片送進來，後端收下後叫剛剛的 analyze() 去讀。

📂 打開這個檔案：app.py，把端點貼在其他 @app.route 的旁邊。

💬 對 AI 這樣說：

請在 app.py 新增一個 /api/vision 端點（POST）：
從 JSON 收 image（dataURL 格式的照片）和 target（目標語言）。
把 dataURL 前綴去掉、base64 解碼成圖片 bytes，
呼叫 providers.analyze(金鑰, target, file_bytes=圖, mime_type=圖的類型)，
最後把 {summary, translation} 回傳成 JSON。

步驟二：AI 會給你這段（貼上去）

貼進 app.py
# app.py
import base64
@app.route('/api/vision', methods=['POST'])
def api_vision():
    data = request.get_json(force=True) or {}
    img = data.get('image','')            # 前端送來的 dataURL
    target = data.get('target','Traditional Chinese (Taiwan)')
    mime='image/jpeg'
    if img.startswith('data:'):
        head,img = img.split(',',1); mime = head.split(':')[1].split(';')[0]
    raw = base64.b64decode(img)           # 還原成圖片 bytes
    key = os.getenv('GEMINI_API_KEY') or os.getenv('GOOGLE_API_KEY')
    res = providers.analyze(key, target, file_bytes=raw, mime_type=mime)
    return jsonify({'ok':True, **res})

步驟二：逐段看懂

「收照片的門」怎麼運作
這幾行在做什麼
@app.route('/api/vision', ...)宣告一個網址，前端就是打這個位址送照片
data.get('image')拿出前端送來的照片（dataURL 文字）
img.split(',',1)dataURL 長得像 data:image/jpeg;base64,xxxx，這裡把「前綴」和「真正的資料」切開
base64.b64decode(img)把文字還原成真正的圖片 bytes
providers.analyze(...)叫步驟一寫好的函式去讀圖
jsonify({'ok':True, **res})把摘要＋翻譯回傳給前端

步驟四：拍完 → 縮圖 → 送出 → 顯示

整章最關鍵的一段前端邏輯
步驟 4／4
🎯 這一步要做什麼：拍完照後，前端先把照片縮小，送到 /api/vision，拿回摘要＋翻譯後顯示出來。

📂 打開這個檔案：static/js/app.js，把這段接在「開相機」那段的下面。

💬 對 AI 這樣說：

照片選好後，先用 canvas 把它縮到最長邊 1568 像素（省流量、變快），
再用 fetch POST 到 /api/vision，body 帶 image（縮好的 dataURL）和 target（目標語言）。
拿到回應後，把 summary 和 translation 顯示在畫面上。

步驟四：AI 會給你這段（貼上去）

貼進 static/js/app.js
cam.onchange = async (e)=>{
  const file = e.target.files[0]; if(!file) return;
  const dataURL = await downscale(file, 1568);   // 縮到最長邊 1568
  const res = await fetch('/api/vision', {method:'POST',
    headers:{'Content-Type':'application/json'},
    body: JSON.stringify({ image:dataURL, target:'Traditional Chinese (Taiwan)' })});
  const d = await res.json();
  showResult(d.summary, d.translation);          // 顯示摘要 + 翻譯
};

💡 downscale() 和 showResult() 是兩個小幫手函式（下一張說明），AI 會一起幫你生出來。實際專案裡它們叫 fileToDownscaledDataURL() 與 renderVision()，作用一樣。

步驟四：逐段看懂

這段每一行在幹嘛
這一行在做什麼
const file = e.target.files[0]拿到使用者剛拍/選的那張照片
await downscale(file, 1568)小幫手①：用 canvas 把照片縮到最長邊 1568，變成 dataURL 文字
await fetch('/api/vision', ...)把縮好的照片送到步驟二做的那道門
await res.json()把後端回來的結果讀成物件 d
showResult(d.summary, d.translation)小幫手②：把摘要與翻譯塞進畫面顯示
🧰 兩個小幫手：downscale()＝把大圖變小圖；showResult()＝把文字放進結果彈窗。你只要對 AI 說「幫我補上 downscale 和 showResult」即可。

為什麼一定要先縮圖？

不縮圖會又慢又失敗

手機拍的照片動輒 4000×3000、好幾 MB。直接上傳會：傳很久、吃流量、還可能因為太大被伺服器擋下來。

縮到最長邊 1568 像素，文字還是看得清楚，但檔案小很多，又快又穩。

✅ 這就是「工程上的小聰明」——在送出前先處理好，體驗差很多。

實測：拍一張外文的東西

成功長這樣

找一張日文/英文菜單或告示，按「📷 拍照翻譯」拍下。幾秒後會跳出「摘要」＋「翻譯」。

實拍：拍下日文場館告示牌，App 幾秒後給出「摘要 + 翻譯」
✅ 預期：上方「📌 摘要」用你的語言講重點，下方「🌐 翻譯」是完整翻譯，還能按「🔊 朗讀」念出來。

🔧 卡住了？本章常見的三個坑

對症下藥
① 拍完很久沒反應／失敗：多半是沒縮圖直接傳大圖。確認 downscale() 有先跑。
② 跳出「（無摘要）」或 JSON 錯誤：沒關 thinking。回步驟一確認 thinking_budget=0。
③ 回 429／額度：Gemini 免費額度到上限了，等一下再試，或換一把金鑰。

動手練習

改一點點，感受差異
  • 把送出時的 target 從 'Traditional Chinese (Taiwan)' 改成 'English'，再拍一次中文招牌，看它翻成英文。
  • 把 downscale(file, 1568) 的 1568 改成 512，拍一張密密麻麻的菜單，看看字太小會不會翻不準——體會「解析度 vs 速度」的取捨。
