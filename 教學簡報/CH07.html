白話觀念：什麼是 STT？

Speech To Text，語音轉文字

STT 就是把「聲音」變成「文字」。你對著手機講話，它幫你打成字。

你可以把它想成「一個很快的聽寫員」：你唸，它一字一字幫你記下來。記好之後，我們再把這段文字交給上一章的翻譯功能。

🔎 好消息：這個聽寫員瀏覽器自己就內建，叫 Web Speech API，不用裝東西、不用申請、不用金鑰。

白話觀念：Web Speech API 是誰提供的

瀏覽器送的免費功能

Web Speech API 是瀏覽器（Chrome、Edge）內建的一組能力，其中的「語音辨識」就是我們要用的聽寫員。它跑在瀏覽器裡，把麥克風收到的聲音送到 Google 的辨識服務轉成文字，全程對我們免費。

✅

支援最好

桌機 Chrome / Edge。本章請優先用它們測試。

⚠️

不一定支援

部分瀏覽器（如某些 Firefox）沒有。所以我們會加「不支援就改用打字」的後備。

白話觀念：語言碼 BCP-47

告訴聽寫員「我要講哪國話」

聽寫員一開始不知道你要講中文還是英文，得先告訴它。方法是給一個「語言碼」，例如：

語言語言碼（BCP-47）
台灣繁體中文zh-TW
英文（美國）en-US
日文ja-JP
韓文ko-KR

步驟一：AI 會給你這段（貼上去）

貼進 static/js/app.js
// app.js — STT（Web Speech API）
// 拿瀏覽器內建的語音辨識；Chrome/Edge 用 webkit 前綴的那個
const SR = window.SpeechRecognition || window.webkitSpeechRecognition;
function sttSupported(){ return !!SR; }

class Recognizer {
  constructor({ bcp, onInterim, onDone, onState }){
    this.bcp=bcp; this.onInterim=onInterim; this.onDone=onDone; this.onState=onState;
    this.active=false; this.rec=null; this.buffer='';
  }
  start(){
    if(!SR){ toast('此瀏覽器不支援語音辨識，請改用文字輸入'); return; }
    if(this.active){ this.stop(); return; }   // 再點一下 = 停止並翻譯
    this.buffer='';                            // 開始新的一段，清空累積
    const rec = new SR();
    rec.lang=this.bcp; rec.interimResults=true; rec.continuous=true;
    rec.onresult=(e)=>{
      let interim='';
      for(let i=e.resultIndex; i<e.results.length; i++){
        const r=e.results[i];
        if(r.isFinal) this.buffer += r[0].transcript;   // 已確定的句段累積
        else interim += r[0].transcript;                // 正在辨識的先暫放
      }
      const shown=(this.buffer+interim).trim();
      if(shown) this.onInterim?.(shown);                // 即時顯示逐字稿
    };
    rec.onerror=(e)=>{
      if(e.error==='no-speech'||e.error==='aborted') return;
      if(e.error==='not-allowed') toast('麥克風權限被拒絕，請允許後重試');
      else toast('語音辨識錯誤：'+e.error);
    };
    rec.onend=()=>{ if(this.active){ try{ rec.start(); }catch{} } // 自動續聽
                    else this.onState?.(false); };
    this.rec=rec; this.active=true;
    try{ rec.start(); this.onState?.(true); }
    catch(e){ toast('無法啟動麥克風'); this.active=false; }
  }
  stop(){
    this.active=false;
    if(this.rec){ try{ this.rec.stop(); }catch{} }
    this.onState?.(false);
    const text=this.buffer.trim();   // 講完了，整段一次交出去翻譯
    this.buffer='';
    if(text) this.onDone?.(text);
  }
}

💡 toast() 是專案裡「跳一條小提示」的小幫手，CH05 已經有了，直接用即可。這裡先不做繁中簡轉繁，那是步驟二專門要補的坑。

步驟一：逐段看懂 (1)

開頭與 start()：怎麼「開始聽」
這幾行在做什麼
const SR = window.SpeechRecognition || window.webkitSpeechRecognition拿瀏覽器內建的聽寫員。有的瀏覽器要加 webkit 前綴，兩個都試，哪個有就用哪個。
if(!SR){ toast(...) }如果這台瀏覽器沒有聽寫員 → 跳提示叫使用者改用打字，不會白畫面。
if(this.active){ this.stop() }關鍵！正在聽的時候「再點一下」= 停止。這就是「點一下開始、再點一下翻譯」。
rec.lang=this.bcp告訴聽寫員「我要講哪國話」（就是前面的 BCP-47 語言碼）。
rec.interimResults=true連「還沒講完、辨識到一半」的字也給我 → 才能即時顯示逐字稿。
rec.continuous=true不要我停一下就自己結束，讓我一直講。

步驟一：逐段看懂 (2)

onresult：逐字稿怎麼即時長出來

onresult 是「聽寫員每聽懂一點就回報一次」的地方。它回來的每一段有兩種狀態：

  • r.isFinal === true：這句確定了，我們把它累加進 this.buffer（永久保留）。
  • r.isFinal === false：這句還在猜，放進暫時的 interim，等下一次可能會被更新。

const shown=(this.buffer+interim).trim()：把「已確定 + 正在猜」接起來顯示，所以你講話時畫面會一個字一個字浮出來。

💡 為什麼要分兩桶？因為停止時我們只送「已確定」的 buffer 去翻譯，正在猜的暫存字不算數，翻譯才乾淨。

步驟一：逐段看懂 (3)

onend「自動續聽」與 stop()

rec.onend：聽寫員講到一半常會自己「斷氣」（辨識引擎每隔一段會自動結束）。這時如果我們還想繼續聽（this.active 還是 true），就馬上再 start 一次，讓它接著聽——使用者完全沒感覺，以為一直在聽。

stop()：使用者「再點一下」時進來。先把 active 設成 false（這樣 onend 就不會再續聽），停掉聽寫員，然後把 buffer 裡的整段文字透過 onDone 交出去翻譯。

✅ 這個設計的精神：邊講邊顯示、講完才翻譯。下一張說明為什麼刻意「講完才翻」。

步驟一：為什麼「說完再翻」而不是邊說邊翻？

逐句模式的用意

我們刻意讓使用者「說完、再點一下停」才整段翻譯，而不是邊說邊翻。理由：

  • 翻譯要完整句子才準；半句半句翻，意思常會跑掉。
  • CH08 會加「朗讀」——如果邊說邊翻邊念，念出來的聲音會被麥克風收回去當成你在講話，造成混亂。講完才翻就沒這問題。
🎯 這就是課程名稱裡「逐句」的意思：一句一句，穩穩來。

步驟二 · 踩過的雷：手機辨識國語，卻回「簡體字」

真實專案遇到的坑，一定要補
步驟 2／4
🐛 症狀：明明 rec.lang 設的是 zh-TW（台灣繁體），在手機（Android / Chrome）上講國語，逐字稿卻跑出「简体字」——例如「谢谢」而不是「謝謝」、「说」而不是「說」。

🔍 原因：手機的語音辨識引擎，國語常常只回簡體，就算你指定 zh-TW 也一樣。這是引擎的行為，我們改不了它，只能在拿到文字後自己轉回繁體。

💡 這一步不是「新功能」，而是「修 bug」。真實專案就是這樣一路踩雷、一路補起來的。

步驟二：AI 會給你這兩個小函式（貼上去）

貼進 // ===== 3. 語音辨識 STT ===== 區

💬 對 AI 這樣說：

手機的國語語音辨識常回傳簡體字，即使 lang 設 zh-TW 也一樣。
幫我用 OpenCC（window.OpenCC）寫一個 toTraditional(text) 把簡體轉繁體，
第一次用到才建立轉換器（lazy），OpenCC 沒載入時原字輸出、不要報錯。
再寫一個 convForBcp(bcp, text)：只有辨識語言是繁體中文(zh-TW/zh-HK)時才轉，其他語言原樣返回。
// 手機（Android/Chrome）的國語辨識常回傳簡體字，即使 lang 設 zh-TW 也一樣。
// 用 OpenCC 把辨識結果轉回繁體；OpenCC 未載入時原字輸出，不影響其他語言。
let _s2t=null, _s2tReady=false;
function toTraditional(text){
  if(!text) return text;
  try{
    if(!_s2tReady){ _s2tReady=true; if(window.OpenCC) _s2t=window.OpenCC.Converter({from:'cn', to:'tw'}); }
    return _s2t ? _s2t(text) : text;
  }catch(e){ return text; }
}
// 只有辨識語言為繁體中文時才需要轉換
function convForBcp(bcp, text){
  return (bcp==='zh-TW' || bcp==='zh-HK') ? toTraditional(text) : text;
}

步驟二：把轉換接進 onresult

改 Recognizer 裡累積文字的那兩行

📂 回到步驟一的 Recognizer，把 onresult 裡累積文字的兩行，各包一層 convForBcp：

rec.onresult=(e)=>{
  let interim='';
  for(let i=e.resultIndex; i<e.results.length; i++){
    const r=e.results[i];
    // 繁中辨識先轉回繁體，避免手機回傳簡體字

    else          interim   += convForBcp(this.bcp, r[0].transcript);

步驟二：逐段看懂

為什麼這樣寫最省事又不踩雷
這幾行在做什麼 / 為什麼
let _s2t=null, _s2tReady=false轉換器第一次用到才建立（lazy）。避免頁面一開就白花時間，也避開 OpenCC 還沒載入好就用。
if(window.OpenCC) ...OpenCC 有載入才建；萬一 CDN 掛了、沒載到，_s2t 保持 null。
return _s2t ? _s2t(text) : text有轉換器就轉；沒有就原字輸出——寧可暫時是簡體，也不要整個當掉。
convForBcp(bcp, text)只有 zh-TW／zh-HK 才轉。講英文、日文的人完全不受影響。
convForBcp(this.bcp, r[0].transcript)在文字「進 buffer 之前」就轉好，所以顯示、翻譯、朗讀全程都是繁體。
✅ 這種「工具沒載到也不崩」的防呆寫法，就是把程式做穩的關鍵。

🔌 接線：這個聽寫器聽到的整段文字，會交給 CH06 做的 translate() 去翻譯——你不用重寫翻譯，只是把「聽到的字」接到「已經會翻的函式」上。這就是疊加：新功能接到舊功能。

// 單人模式的聽寫器：設定「顯示」「翻譯」「按鈕變色」三件事
const sRecognizer = new Recognizer({
  bcp: byId(cfg.s_langA).bcp,                      // 起始語言碼（來源語）
  onInterim: (t)=> setResult(sResult, t, true),    // 逐字稿即時顯示
  onDone: async (t)=>{                             // 按停後才進來：整段翻譯
    const a=$('s_langA').value, b=$('s_langB').value;
    try{
      setResult(sResult, '翻譯中…', true);
      const out = await translate(t, a, b);         // 沿用 CH06 的翻譯
      setResult(sResult, out);
      speak(out, byId(b).bcp);                       // 自動朗讀（CH08 深講）
      pushHistory(t, out);
    }catch(e){ toast(e.message); setResult(sResult, '（翻譯失敗）'); }
  },
  onState: (on)=> toggleMic($('s_mic'), on),        // 聆聽時按鈕變色
});

  2. 來源語言選「CHT-台灣繁體」，點「🎤 點一下開始說話」。
  3. 瀏覽器跳出「允許使用麥克風？」→ 按 允許。
  4. 說一句中文（如「你好，很高興認識你」），逐字稿會即時浮現。
  5. 再點一下麥克風停止 → 整段被翻成目標語言、並自動念出。

🔧 卡住了？本章常見的坑

對症下藥
① 完全沒反應：多半是瀏覽器不支援。請用 Chrome / Edge；本機用 http://127.0.0.1 沒問題，但正式上線一定要 HTTPS，否則瀏覽器不給用麥克風。
② 權限被拒：網址列左邊「🔒 鎖頭」→ 網站設定 → 把「麥克風」改成允許 → 重整頁面。
③ 逐字稿是簡體字：OpenCC 沒載到（CDN 被擋／打錯網址），或忘了用 convForBcp 包。回步驟二檢查那行 <script> 與 onresult。
④ 辨識成別的語言：來源語言選錯，或 rec.lang 沒吃到最新選項。確認點麥克風時有 rec.bcp = byId($('s_langA').value).bcp。

🔧 除錯：手機上不太穩？

STT 的先天限制與備案

瀏覽器內建 STT 在手機上有時會斷斷續續、或辨識較慢——這是引擎的限制，不是你程式寫錯。

  • 短句、講清楚、環境安靜，成功率高很多。
  • 真的要「低延遲、邊講邊翻」，可搭配後面 CH09 的 Gemini Live 即時模式（畫面上「即時」那顆選項），走串流、體驗更順。
🧭 本章的「逐句」模式免金鑰、支援所有翻譯供應商，是最穩的入門法；即時模式是進階選項，兩者並存。

動手練習

改一點點，感受差異
  • 把 Recognizer 裡的 rec.continuous 改成 false，講一長句，觀察它是不是講到一半就自己停了——體會 continuous=true 為什麼重要。
  • 暫時把 onresult 裡的 convForBcp(this.bcp, …) 改回 r[0].transcript（拿掉簡轉繁），在手機上講國語，看它是不是又跑出簡體字——親眼見證步驟二那個坑。
  • 把來源語言換成「Japanese-日本語」，唸一句日文，看逐字稿與翻譯——確認語言碼真的有換到。
