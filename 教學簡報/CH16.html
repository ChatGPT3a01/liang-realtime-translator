這一章要做出什麼

網頁 → 可安裝的 App

我們要把網站升級成 PWA。做完後，手機打開你的網址，瀏覽器選單會出現「加到主畫面」；裝好後主畫面多一顆圖示，點它就全螢幕開，沒有網址列，跟商店下載的 App 幾乎一樣。

🎁 本章小成品

一個能「加到主畫面」的翻譯 App——圖示、名稱、全螢幕都有；而且開得比較快，靜態畫面弱網也打得開。

白話觀念：什麼是 PWA

Progressive Web App＝可安裝的網頁

PWA 就是「穿了 App 外衣的網頁」。你不用把它上架 App Store／Google Play，使用者只要用瀏覽器打開網址，就能把它加到手機主畫面，之後點圖示全螢幕開啟。

要變成 PWA，只需要幫網站補兩樣東西：

📇

manifest

App 的「身分證」：告訴手機它叫什麼、用哪個圖示、什麼顏色、要不要全螢幕。

⚙️

Service Worker

躲在背景的小管家：攔截網路請求、做快取，讓 App 開得快、靜態畫面能離線。

🔌 接線：PWA 不改動任何功能，只是幫網頁「多穿一層外殼」——manifest 讓它能加到主畫面、Service Worker 在背景管快取。翻譯、語音、拍照、問答等 App 本身的邏輯完全不動，一行都不用改。

先看全貌：使用者裝 App 時發生什麼事

看懂這條路，後面每一步就知道自己在做哪一段
  1. 手機瀏覽器打開你的網址，載入 index.html
  2. 網頁裡的 <link rel="manifest"> 讓瀏覽器讀到 manifest.json（App 的名稱、圖示、顏色）
  3. 網頁的 JS 呼叫 navigator.serviceWorker.register('/sw.js') 註冊背景管家
  4. 瀏覽器判斷「這是合格的 PWA」→ 選單出現「加到主畫面」
  5. 使用者點下去 → 主畫面多一顆圖示
  6. 之後點圖示 → 全螢幕開啟，Service Worker 在背景幫忙加速與快取
👉 這一章就是照著這條路，把 manifest、sw.js、以及「連結＋註冊」三塊補齊。

先說清楚：PWA 能離線到什麼程度

別誇大——這關係到使用者信任

很多人以為「PWA＝完全離線可用」，這是誤會。我們要誠實界定：

情境離線時
App 的殼（畫面、按鈕、樣式、圖示）✅ 打得開（因為被快取了）
翻譯、拍照、天氣、問答（要打 Gemini／API）❌ 不行，這些一定要連網
⚠️ 所以正確說法是：「殼與靜態資源可離線開啟，但翻譯等功能需連網」。教學或宣傳時千萬別寫成「離線也能翻譯」——那是做不到的，會失去使用者信任。

這幾行在做什麼

步驟二 · 為什麼用「網路優先」而不是「快取優先」

這個選擇決定使用者會不會看到舊版

Service Worker 有兩種常見策略：

🕸️

快取優先

先給快取、開超快，但你改了程式重新部署，使用者一直看到舊版（被快取黏住）。

🌐

網路優先（我們用這個）

先上網拿最新，順便更新快取；沒網路才用快取。不會被舊版卡住，離線也還有殼可用。

✅ 翻譯 App 常改常上線，「使用者總是看到最新版」比「快 0.2 秒」重要太多，所以選網路優先。

「叫小管家上工」怎麼寫

if ('serviceWorker' in navigator)：先問「這個瀏覽器支不支援 Service Worker？」——支援才做，不支援也不會壞（很老的瀏覽器就跳過，網站照常當普通網頁用）。

.catch(e => console.warn(...))：萬一註冊失敗，只在 Console 印個警告，不影響翻譯等主要功能——「優雅地失敗」。

🕳️ 全章最常見的坑：改了東西，手機卻看不到新版

被舊快取黏住了

裝了 PWA 之後，最讓人抓狂的情況是：你明明改好程式、重新部署了，手機打開卻還是舊畫面。原因是 Service Worker 手上還握著舊快取。

✅ 標準解法（記起來）：
① 把 static/sw.js 的 CACHE 版本號 +1（例：'liang-translate-v11' → 'liang-translate-v12'）。
② 重新部署後，在手機／瀏覽器重整兩次（第一次讓新 SW 接手清舊快取，第二次才吃到全新版）。
🔧 版本號沒 +1 → 舊 SW 覺得「我這版沒變」，就不會清快取，你永遠看到舊的。所以每次改前端就 +1，養成習慣。

版本號 +1 到底發生什麼

看懂就不會再被卡住

瀏覽器判斷「Service Worker 有沒有更新」，靠的是 sw.js 這個檔內容變了沒。你把 CACHE 從 v11 改成 v12，檔案內容就變了 → 瀏覽器認定「有新版」→ 觸發 install → skipWaiting 立即接手 → activate 時清掉所有舊快取。

👉 這段程式改用本章開頭的 🧩 生成器 產生（複製提示詞 → 送出 → 採用），不必手打。
🧹 想手動強制清乾淨（電腦測試時）：F12 → Application → 左邊 Service Workers 按 Unregister，再到 Storage 按 Clear site data，然後重整。

裝好之後：怎麼確認離線界線

自己驗證一次，才敢對別人說

裝好後做個小實驗，親眼看見「殼可離線、功能要連網」：

  1. 連網狀態先開一次 App（讓殼被快取起來）。
  2. 開啟手機飛航模式（斷網），再點圖示打開 App。
  3. 你會看到：畫面、按鈕、樣式都在（殼載得出來）。
  4. 但按「翻譯」→ 會失敗或轉圈——因為它要連 Gemini，沒網路做不到。
✅ 這正是我們一開始講的：殼可離線開，翻譯要連網。實測過，你就能誠實地向使用者說明。

🔧 卡住了？本章常見的四個坑

對症下藥
① 選單沒有「加到主畫面」：多半缺 HTTPS（正式網址要 https），或 manifest 沒載到／格式錯。F12 → Application → Manifest 看有沒有紅字。
② 改了程式手機還是舊版：把 sw.js 的 CACHE 版本號 +1，重新部署，重整兩次。（本章第一大坑）
③ SW 註冊失敗（Console 紅字）：確認 app.py 有 /sw.js 路由、且回應帶 Service-Worker-Allowed: /；路徑要是根路徑 /sw.js。
④ 圖示破圖或沒圖：static/icon-192.png、icon-512.png 真的存在嗎？manifest 裡的 src 路徑要對得上。

動手練習

改一點點，感受變化
  • 把 manifest 的 theme_color 改成別的顏色（例 #0d7d5a 綠），重整後看手機狀態列／啟動畫面顏色變了。記得 sw.js 版本號 +1，不然看不到新色。
  • 把 display 從 "standalone" 改成 "browser"，重裝一次，感受「有沒有全螢幕、有沒有網址列」的差別，再改回 standalone。
  • 故意不改版本號，改一下 index.html 文字，重整——體會「被舊快取黏住」的真實感；再把版本號 +1、重整兩次，看它換新。
