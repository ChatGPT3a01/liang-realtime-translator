為什麼要「供應商抽象層」？

providers.py 的用意

我們不直接在 app.py 裡呼叫 Gemini，而是獨立出一個檔 providers.py，把「呼叫哪家 AI」包起來。

為什麼多這一層？因為翻譯引擎將來可能想換——今天用 Gemini，明天想用 OpenAI 或 Groq。有了這層，換引擎只要改 providers.py，app.py 和前端完全不用動。

用一個比喻理解

抽象層 = 萬用轉接頭

想像你的電器（app.py）要插電。不同國家插座（Gemini／OpenAI）形狀不一樣。

providers.py 就是萬用轉接頭：電器永遠插同一個孔，轉接頭負責對接不同插座。要換國家，只換轉接頭。

🔌

providers.py（完整可複製）

新建這個檔、貼上
# providers.py
import os
from google import genai
from google.genai import types

def build_prompt(source, target):
    return (f"You are a professional translation engine. "
            f"Translate from {source} into {target}. "
            "Output ONLY the translated text — no explanations.")

def translate(data):
    text = (data.get('text') or '').strip()
    source = data.get('source') or 'auto'
    target = data.get('target') or 'English'
    if not text:
        return {"ok": False, "error": "empty text"}
    key = os.getenv('GEMINI_API_KEY') or os.getenv('GOOGLE_API_KEY')
    if not key:
        return {"ok": False, "error": "找不到 Gemini 金鑰"}
    try:
        client = genai.Client(api_key=key)
        resp = client.models.generate_content(
            model='gemini-3.5-flash',
            contents=text,
            config=types.GenerateContentConfig(
                system_instruction=build_prompt(source, target),
                temperature=0.3))
        return {"ok": True, "translation": (resp.text or '').strip()}
    except Exception as e:
        return {"ok": False, "error": f"{type(e).__name__}: {e}"}

逐段看懂 (1)：開頭與指令

import 與 build_prompt

import os：讓程式能讀「環境變數」（金鑰放那裡）。

from google import genai：載入 Gemini 的 SDK。

build_prompt(source, target)：組一句「給 AI 的指令」，告訴它「把 source 語言翻成 target，而且只輸出翻譯、不要囉嗦」。這句話品質，直接決定翻譯乾不乾淨。

💡「system 指令」就是給 AI 的角色設定與規則。寫得越明確，AI 越聽話。

逐段看懂 (2)：處理輸入

translate 函式的前半

data.get('text')：從前端送來的資料拿出「要翻的文字」。.strip() 去掉頭尾空白。

if not text:：如果沒文字，直接回一個錯誤，不浪費一次 API 呼叫。

key = os.getenv(...)：讀金鑰。先找 GEMINI_API_KEY，沒有再找 GOOGLE_API_KEY（兩個名字都支援，較保險）。

✅ 這種「先檢查、再動作」的寫法，能擋掉很多低級錯誤，是好習慣。

逐段看懂 (3)：呼叫 Gemini

真正翻譯的那幾行

client = genai.Client(api_key=key)：拿金鑰建立一個「Gemini 連線客戶端」。

client.models.generate_content(...)：請 Gemini 產生回應。三個重點參數：

  • model='gemini-3.5-flash'：用哪個模型（3.5 flash 快又便宜）。
  • contents=text：要處理的內容（使用者的文字）。
  • system_instruction=...：我們的翻譯規則。

resp.text：Gemini 回的文字，就是翻譯結果。

逐段看懂 (4)：錯誤處理

try / except 的用意

try: ... except Exception as e:：把「可能出錯」的動作包起來。萬一呼叫失敗（沒網路、金鑰錯、額度到），程式不會整個當掉，而是回一個帶錯誤訊息的結果。

return {"ok": False, "error": ...}：把錯誤原因帶回前端，方便你看到「到底哪裡錯」。

🛡️ 有 try/except，使用者看到的是「翻譯失敗」而不是白畫面。這叫「優雅地失敗」。

在 app.py 加 /api/translate

把 providers 接進來

📂 打開這個檔案：app.py，找到 # ===== 2. 翻譯 API ===== 這一區，把這段貼在它下面。💡 上面的 from flask import request、import providers 放在檔案最上面的 import 區。

# app.py 最上面的 import 區補這兩行
from flask import request
import providers

# ===== 2. 翻譯 API =====（把下面這段貼在這一區）
@app.route('/api/translate', methods=['POST'])
def api_translate():
    data = request.get_json(force=True, silent=True) or {}
    result = providers.translate(data)
    return jsonify(result), (200 if result.get('ok') else 400)

逐段看懂：這段 app.py

後端多了一個「窗口」

@app.route('/api/translate', methods=['POST'])：開一條新路由，而且限定用 POST（帶資料的請求）。

request.get_json(...)：把前端送來的 JSON 資料拿出來。

providers.translate(data)：交給我們剛寫的翻譯函式。

return jsonify(result), (200 if ... else 400)：成功回 200、失敗回 400，並把結果轉成 JSON 送回前端。

小補充：POST 與 JSON

兩個常聽到的名詞
📮

POST

一種「帶著資料去請求」的方式（相對於只是「看網頁」的 GET）。翻譯要把文字送過去，所以用 POST。

📦

JSON

一種前後端溝通的資料格式，長得像 {"text":"你好"}。輕巧、通用。

前端：呼叫翻譯

app.js 的 translate 函式

📂 打開這個檔案：static/js/app.js，找到 CH05 建立的 // ===== 2. 翻譯 ===== 這一區，把這段貼在它下面。💡 這就是骨架分區的好處——不用想「貼哪」，照著區塊名字放就對了。

// static/js/app.js　// ===== 2. 翻譯 =====
async function translate(text, source, target){
  const res = await fetch('/api/translate', {
    method:'POST',
    headers:{'Content-Type':'application/json'},
    body: JSON.stringify({ text, source, target })
  });
  const data = await res.json();
  if(!data.ok) throw new Error(data.error);
  return data.translation;
}

逐段看懂：前端 fetch

前端怎麼「打電話」給後端

fetch('/api/translate', {...})：向後端發出請求。method:'POST' + body: JSON.stringify(...) 把文字打包送過去。

await res.json()：等後端回覆，並把 JSON 解析成物件。

if(!data.ok) throw ...：後端說失敗就丟出錯誤，讓呼叫它的地方顯示「翻譯失敗」。

💡 async／await 是處理「要等一下才有結果」的寫法（等 AI 回覆）。

語言代碼 → 給模型的名稱

這張對照表哪來的

選單的值是代碼（如 zh-TW），但要告訴 Gemini「翻成什麼」得用英文名稱。做一張對照表：

// app.js — 語言代碼對應給模型看的名稱
const NAME = {'zh-TW':'Traditional Chinese','en':'English',
              'ja':'Japanese','ko':'Korean'};

接到「翻譯」按鈕

讓 send 真的動起來

📂 一樣放在 static/js/app.js 的 // ===== 2. 翻譯 ===== 這一區裡（接在剛剛的 translate() 後面）。

🔌 接線：這顆「打字翻譯鈕」按下去，就是呼叫上面的 translate()。之後 CH07 的麥克風（聽到的字）、CH10 的面對面都會重用同一個 translate()——不用重寫，把不同來源的文字接到「已經會翻的函式」上就好。這就是疊加：新功能接到舊功能。
// 這裡的 langA、langB、result 是 CH05 用
// document.getElementById 取得的元素，沿用即可
document.getElementById('send').onclick = async () => {
  const text = document.getElementById('text').value.trim();
  if(!text) return;
  result.textContent = '翻譯中…';
  try{
    const out = await translate(text, NAME[langA.value], NAME[langB.value]);
    result.textContent = out;
  }catch(e){
    result.textContent = '（翻譯失敗）';
    alert(e.message);
  }
};

除錯 1：找不到金鑰

最常見的第一個坑

症狀：結果框顯示「翻譯失敗」，錯誤訊息含「找不到 Gemini 金鑰」。

檢查清單：

  • .env 是否在專案根目錄（跟 app.py 同一層）？
  • .env 裡是否有 GEMINI_API_KEY=AIza...？
  • app.py 最上面是否有 load_dotenv()？
  • 改了 .env 後，伺服器重啟了嗎？

除錯 2：回 429 額度錯誤

額度或上限到了

症狀：錯誤訊息含 429 或 RESOURCE_EXHAUSTED。

原因：Gemini 免費額度用完，或你設的「月支出上限」到了。

解法：到 ai.studio/spend 查看並調整上限；或稍等額度重置。課堂練習量通常不會遇到。

除錯 3：翻譯多了廢話

出現「翻譯結果是：…」之類

症狀：翻譯前後多了「這句話的意思是」「Here is the translation:」等多餘文字。

解法：加強 build_prompt 的指令，例如再補一句 "Do not add any prefix or explanation."。指令越明確，輸出越乾淨。

🎯 這就是「提示詞工程」——同一個模型，指令不同，結果差很多。

除錯 4：前端呼叫沒反應

按了按鈕結果框不動

檢查：按 F12 打開瀏覽器 Console 看有沒有紅字錯誤。

  • 404：後端沒有 /api/translate，確認 app.py 加對、重啟了。
  • langA is not defined：前端變數沒接上，確認 CH05 的元素有取得。
  • 什麼都沒印：按鈕的 onclick 沒綁上，確認 id 是 send。

加分：抽象層的威力

未來想換 OpenAI？

因為有 providers 這層，要多支援 OpenAI／Groq 等「OpenAI 相容」服務，只要在 providers.py 加一個分支，用 requests 打它們的 /chat/completions。

app.py 和前端完全不用改——這就是一開始多花力氣做抽象層的回報。
